# 文本化无人机-地面车辆 Re-ID 数据集 制作动机

## 1.  无人机-地面车辆 Re-ID 的动机

随着低空经济、无人机巡检和空地协同感知系统的发展，车辆重识别的应用边界正在从传统地面多摄像头网络扩展到无人机与地面摄像头之间的异构平台检索。传统车辆 Re-ID 主要关注固定卡口、路侧摄像头或停车场监控之间的车辆匹配，而空地跨视角车辆 Re-ID 进一步要求系统在 UAV 俯视图像与地面近景图像之间建立同一车辆身份关联。以 CVnet/CVPair 为代表的真实空地车辆数据提供了 a2g 与 g2a 双向检索协议，使研究者能够系统评估从空中视角检索地面车辆、以及从地面视角检索空中车辆的能力。

空地车辆 Re-ID 的核心难点来自极其严苛的视点差异。地面视角通常能够观察车辆的正面、尾部或侧面，在现实监控中还可能包含车牌、中网、大灯、挡风玻璃标志、轮毂和车身局部纹理等细节；而无人机俯视或斜俯视视角主要呈现车顶、天窗、引擎盖、后备箱顶和整体轮廓。两类视角之间存在显著的信息不对称，同一车辆在两个视角中的可见部件高度不一致，真正可对齐的局部区域可能非常有限。同时，无人机高度、俯仰角和方位角变化会造成车辆几何形状的压缩、拉伸和拓扑变形，使传统基于二维外观相似度的 CNN/ViT 表征难以直接建立稳定映射。在小目标、遮挡、低分辨率和强透视变化条件下，模型还容易依赖背景、停车位置或相机域特征，而不是学习车辆本身的跨视角身份线索。

现有主流车辆 Re-ID 数据集难以充分覆盖这一问题。多数数据集建立在地面固定摄像头或路侧卡口网络上，即使包含一定高度差或俯仰角，也更接近地面水平视角的延伸，而非真正的 UAV-to-ground 跨平台匹配。CVnet/CVPair 的价值在于提供了真实空地同 ID 图像，并保留空到地、地到空两个检索方向；但其采集特性也需要被明确说明：车辆通常处于静止状态，同一 ID 多张图像往往来自相同或相近场景，背景变化不足，空中侧则提供多个角度和帧来覆盖俯视姿态变化。因此，该数据更适合作为研究空地强视角差异、多角度俯视匹配和语义桥接的核心基准，而不应被表述为已经完整覆盖跨地点、跨时间、跨背景的开放世界车辆追踪。

从应用角度看，空地跨视角车辆检索对应明确的现实需求。例如，交警无人机在空中巡查时发现肇事逃逸或违章车辆，需要联动地面卡口和路侧摄像头进行轨迹回溯；在搜救、缉私、园区巡逻或复杂地形巡检中，无人机可先在大范围内锁定可疑车辆，地面警车、执法车辆或移动端摄像头再在车流中完成确认与拦截。在远距离、高空、夜间、反恐或军事侦察等场景下，车牌往往不可见、模糊或因隐私合规被遮挡，系统必须依赖车顶结构、天窗比例、行李架、天线、车身轮廓、颜色、侧面细节和时空上下文完成匹配。与此同时，并非所有任务都要求精确到唯一车辆 ID；在停车场、港口、园区和应急调度中，车型级检索、车辆类型统计和属性级粗筛同样具有实际价值，例如统计 SUV、轿车、厢式车和货车比例，或先筛选“白色 SUV”“深色厢式车”等候选再进行人工复核。

构建文本化空地车辆 Re-ID benchmark 的学术价值在于，它不仅补充了一个新数据集，还能推动新的跨视角建模问题。强视角不对称要求研究者发展细粒度部件对齐、几何先验、3D 隐式表征、视角归一化和跨模态语义对齐方法，而不能只依赖普通地面车辆 Re-ID backbone。车辆作为刚性物体具有稳定结构和方向性，这与行人 Re-ID 中人体姿态变化不同，也为建模车顶、车头、车尾、侧面之间的几何约束提供了独特空间。同时，同款同色车辆在高空视角下极易混淆，如何结合地面侧局部细节、文本属性、时空上下文和背景泄漏分析进行判别，是具有研究价值的挑战。进一步引入自然语言描述和结构化属性，可以把颜色、粗粒度车型、车顶结构、可见部件、遮挡状态和场景上下文显式记录下来，用于连接空地不对称视角并解释模型失败原因；而 AirSim 可控扩展则可补足真实 CVPair 中背景、天气、光照、高度和俯仰角变化不足的问题，为合成到真实迁移、可控消融和鲁棒性评估提供支撑。


## 2. 在传统Re-ID数据集上加入文本的动机

### 2.1 填补空地车辆 Re-ID 的文本 benchmark 缺口

目前公开车辆文本检索数据集以地面多摄为主，例如 CityFlow-NL；无人机车辆 Re-ID 数据集多数只有图像、ID、轨迹或结构化属性，缺少自然语言描述。严格同时具备“无人机/高位视角、地面视角、同一车辆 ID、自然语言文本”的公开 benchmark 非常少。

因此，本项目的直接动机是构建一个文本增强的空地车辆 Re-ID benchmark：在真实 CVnet/CVPair 空地同 ID 数据上补充 ID 级自然语言描述和图像级结构化属性，使研究者不仅能评估 image-to-image Re-ID，还能评估 text-to-image、image-to-text 和 text-guided cross-view Re-ID。


### 2.2 真实应用中查询往往来自语言，而不是完美图像

在交通管理、园区巡检、应急搜寻和空地协同感知中，实际查询不一定是一张质量很高的车辆图片。操作者更常见的输入可能是：

- “找一辆白色厢式货车，车顶较平，出现在停车场边缘。”
- “搜索从无人机画面中看到的蓝色 SUV，在地面摄像头中是否出现。”
- “定位一辆车顶带天窗、深色车窗的白色轿车。”
- “统计港口堆场中货车、厢式车和轿车的大致数量，不要求精确到单车 ID。”
- “在园区停车区检索一类疑似 SUV 的车辆，先做粗筛，再交给人工确认具体实例。”

纯视觉 Re-ID 假设查询图像可用且足够清晰，但 UAV 小目标、遮挡、运动模糊和俯视角会让图像查询本身变弱。自然语言可以把人的先验观察转成可检索的语义条件，提升人机协同检索的实用性。

因此，数据集不仅是“给图片加 caption”，而是在定义一个更接近真实工作流的检索任务：空中或地面观测触发，人类或系统产生文本描述，再在另一视角中检索同一车辆；在牌照不可见或任务只要求统计时，也可以退化为车型、车身类别和颜色属性层面的粗粒度检索。

### 2.3 语言可以作为跨视角稳定语义桥梁

空地车辆 Re-ID 的核心困难不是普通外观匹配，而是强视角不对称：

- 地面侧可见车头、车尾、车窗、轮毂、侧面轮廓。
- 空中侧可见车顶、天窗、车身整体形状、停放姿态。
- 同一车辆在不同视角下的局部纹理几乎不重合。

视觉模型容易过拟合背景、相机高度、停车位、道路纹理等非身份因素。文本标注可以显式抽取相对稳定的身份语义，例如颜色、车型、车身形状、车顶结构、厢体形态、特殊涂装和可见标识，从而提供一种跨视角对齐信号。

关键不是期待文本描述所有视觉细节，而是让文本约束模型关注“跨视角仍可复用的身份属性”。这可以支持以下研究问题：

- ID 级描述是否比逐图 caption 更适合跨视角 Re-ID？
- 结构化属性文本和自然语言描述哪个更有效？
- 文本监督是否能减少模型对背景和相机域的依赖？
- 当 UAV 图像目标很小时，文本能否补充视觉特征的不确定性？

### 2.4 CVPair 暴露了背景捷径与视角捷径问题

CVnet/CVPair 的真实价值在于提供了成对的空地同 ID 图像，但其静止采集和相近场景也会带来背景捷径风险。如果同一车辆 ID 经常绑定同一停车位、地面纹理、车库柱体、邻车布局或道路边界，模型可能不是在学习车辆身份，而是在利用场景上下文完成检索。空中侧多角度图像也可能形成视角捷径：模型记住某一固定停车姿态或俯视背景，而不是学习跨视角可迁移的车辆属性。

这反而构成了文本化 benchmark 的一个重要研究动机。文本标注应把背景记录为 `scene_context`，用于错误分析和背景泄漏检测，但身份描述本身应聚焦车辆可迁移属性，例如颜色、粗粒度车型、车身轮廓、车顶结构、车窗形态、厢体形态和显著外观标识。这样可以把“模型是否依赖背景”从隐性风险变成可测量问题，也能解释为什么需要 AirSim 或其他扩展数据系统控制场景、位置和背景变化。

### 2.5 牌照遮挡使车型级检索成为合理应用

CVnet/CVPair 对牌照进行了遮挡或弱化处理，这符合隐私和公开数据集发布要求，但也削弱了实例级唯一身份线索。在真实部署中，车牌也可能因为角度、距离、遮挡、运动模糊、过曝或合规脱敏而不可用。此时，如果仅依靠颜色、车型、车顶、轮廓和局部外观，完全相同或高度相似的车辆实例可能发生混淆，尤其是同款、同色、同车身类别车辆集中出现的停车场、园区和港口场景。

因此，空地车辆检索不应只被表述为“必须精确找到同一辆车”。在一些应用中，车型级或属性级检索本身就是合理目标：例如统计停车场中轿车、SUV、货车、厢式车的比例；在港口、物流园区或应急救援中估计可通行、可调度或占道车辆类型；在 UAV 巡查中先用“白色 SUV”“深色厢式车”等属性粗筛候选，再由人工或更强模型做实例确认。实例级 Re-ID 仍是 benchmark 的核心目标，但车型级检索、属性检索和区域车辆构成统计可以作为更稳健的补充任务，特别适合解释牌照遮挡后的应用边界。

### 2.6 CVnet 的质量问题本身需要文本化 QA 机制

CVnet 数据中存在小目标、遮挡、邻车干扰、目标边界不稳定和格式不统一等问题。传统 Re-ID benchmark 通常只提供 ID 和图片路径，难以表达这些样本级不确定性，导致模型训练和错误分析都比较粗糙。

本项目可以把文本标注扩展为 QA 机制：

- ID 级描述记录稳定身份属性，减少同一 ID 多张相似图的重复 caption。
- 图像级属性记录视角、可见部件、遮挡、目标大小、背景干扰和 `uncertain` 字段。
- `qa_status` 标记样本是自动通过、人工复核、修正还是丢弃。
- 对低分辨率和遮挡样本不强行生成确定描述，避免 VLM 幻觉。

这种设计让数据集比普通 caption 数据更可控，也更适合严肃 benchmark：模型失败时可以分析是视角问题、尺度问题、文本歧义、背景泄漏，还是标注不确定性导致。

### 2.7 AirSim 合成数据提供可控变量，而不是替代真实数据

真实 CVnet 是核心，但它的视角、天气、光照、车型和背景分布不可控。AirSim 的价值在于补充可控变量：

- 固定同一车辆 ID 的颜色、车型、材质和特殊标识。
- 系统采集地面近景、低空斜视、中高空俯视和近垂直俯视。
- 控制天气、昼夜、相机高度、俯仰角、场景和背景干扰。
- 保存相机位姿、车辆位姿、bbox、mask、depth 和真值属性。

建议 Pilot 规模为 300 个合成 ID，每个 ID 8 个视角，共 2,400 张图像，其中 2 张地面侧、6 张无人机/高位侧。AirSim 数据应定位为 controlled augmentation 和 stress test，而不是声称替代真实世界数据。

这个动机的论文价值在于：可以把“合成数据是否有帮助”从经验性扩充变成可分析实验。例如：

- 真实数据训练 vs 真实+AirSim 训练。
- 只加合成图像 vs 加合成图像和真值文本。
- 不同高度、俯仰角、天气下的文本-视觉对齐稳定性。
- 合成预训练是否提升真实 CVnet 上的 `a2g/g2a` 检索。

### 2.8 数据集可以支持多任务 benchmark，而不是单一 Re-ID 指标

如果只做视觉 Re-ID，本项目的新意会偏弱，因为 CVnet 本身已经提供图像和 ID。文本标注的价值在于扩展 benchmark 任务：

| 任务 | 查询 | Gallery | 评估指标 | 研究意义 |
| --- | --- | --- | --- | --- |
| Visual a2g Re-ID | UAV image | Ground images | mAP, Rank-1/5/10 | 复现传统空到地检索 |
| Visual g2a Re-ID | Ground image | UAV images | mAP, Rank-1/5/10 | 复现地到空检索 |
| Text-to-ground retrieval | ID/text description | Ground images | mAP, Recall@K | 模拟语言找地面车辆 |
| Text-to-UAV retrieval | ID/text description | UAV images | mAP, Recall@K | 模拟语言找空中车辆 |
| Text-guided a2g/g2a | Image + text | Cross-view images | mAP, Rank-K | 检验文本是否帮助跨视角匹配 |
| Attribute retrieval | Color/type/orientation | Images | Recall@K, attribute accuracy | 检验结构化属性是否有效 |
| Vehicle type retrieval | Vehicle type / body category | Images or regions | Recall@K, type accuracy | 支持车型统计、粗筛和牌照不可用场景 |
| Synthetic-to-real transfer | AirSim image/text | CVnet images | mAP, Rank-K | 检验合成数据迁移价值 |

这样的 benchmark 更容易形成论文贡献：不是只发布标注，而是发布任务定义、划分协议、baseline 和系统性的消融。

### 2.9 VLM 辅助标注降低成本，但必须有可审计流程

全人工逐图写自然语言成本较高，且标注者风格不一致。利用 Qwen3-VL 等 VLM 进行初始标注，再由人工抽查和修正，可以显著降低成本。但车辆 Re-ID 文本标注不能简单依赖 VLM caption，需要采用可审计流程：

- 每个 ID 输入多张代表图，生成一条稳定 ID 描述。
- 每张图保留结构化属性，记录视角下实际可见部件。
- 禁止自动生成品牌、具体型号、车牌号和不可见细节，也不把这些字段作为身份特征。
- 允许记录粗粒度 `vehicle_type`，例如 sedan、SUV、van、truck、bus，用于属性检索和类型统计。
- 背景只进入 `scene_context`，不得写成车辆身份特征。
- 低光、小目标、遮挡、颜色难辨时允许 `uncertain`。
- 保存 prompt、模型版本、API usage、人工修改记录和 QA 状态。

这个流程本身也是可投稿的工程贡献：它说明如何从无文本车辆 Re-ID 数据构建高质量文本 benchmark，而不只是生成一批不可复现 caption。
